# 留学规划 Agent App（开发迭代计划）

## 1. 产品目标与范围

### 1.1 核心目标
构建一个「对话式留学规划 Agent」，能够：
1. 读取并结构化用户背景信息（如本科学校、均分/GPA、预算、目标国家、专业方向、语言成绩等）。
2. 对缺失信息进行追问补全。
3. 基于规则 + 检索 + 大模型推理，生成可执行留学方案。
4. 输出院校申请成功率分层（冲刺/匹配/保底）、申请时间轴与行动清单。

### 1.2 MVP 范围（第一阶段）
- 支持硕士申请场景（先聚焦中美英港新澳）。
- 支持单用户单次会话规划。
- 生成结构化报告（可读文本 + JSON）。
- 暂不覆盖文书自动写作、签证全流程、多轮长期托管。

## 2. 用户输入与信息补全设计

### 2.1 核心采集字段（建议）
- 基础画像：
  - 学历层次（本科/专科/高中）
  - 本科学校、专业、均分/GPA、年级/毕业时间
- 语言与考试：
  - IELTS/TOEFL/GRE/GMAT（可为空）
- 申请偏好：
  - 目标国家/地区
  - 目标专业方向
  - 入学季（如 2027 Fall）
- 资源约束：
  - 年预算（统一为 万人民币/年）
  - 可接受学制（1年/2年等）
- 风险与偏好：
  - 名校优先/就业优先/移民优先
  - 是否接受跨专业

### 2.2 缺失追问策略
- 优先级追问顺序：
  1. 必填：学历背景、GPA、预算、目标国家、目标专业、入学季
  2. 强建议：语言成绩、科研/实习、偏好约束
- 一次追问最多 2~3 项，避免用户疲劳。
- 对模糊值做标准化追问：
  - “预算 30 万”追问是总预算还是每年预算。
  - “成绩还行”追问具体 GPA/均分区间。

---

## 3. 方案输出规格（产品结果定义）

最终输出应包含以下模块：
1. **用户背景摘要**：系统理解后的标准画像。
2. **可行国家/地区建议**：按预算、录取难度、就业/移民政策做解释。
3. **院校清单与概率分层**：
   - 冲刺（20%-40%）
   - 匹配（40%-70%）
   - 保底（70%+）
4. **项目推荐理由**：课程、排名、地理位置、费用、就业表现。
5. **申请时间轴**（按月份）：考试、选校、文书、推荐信、网申、面试、签证。
6. **差距诊断与提升建议**：语言、科研、实习、先修课、文书策略。
7. **风险提示**：政策变化、预算不足、DDL 风险。
8. **行动清单（TODO）**：未来 1-2 周可执行事项。

### 4.1 关键模块职责
- `extractor_agent`：把自然语言转成结构化 Profile，并做单位归一（GPA/预算）。
- `clarifier_agent`：根据缺失字段生成高质量追问。
- `planner_agent`：综合用户画像 + 检索结果生成方案。
- `evaluator_agent`：检查结果是否覆盖“国家+院校概率+时间轴+风险提示”。
- `retrieval`：连接学校库与政策库，提供事实支撑，减少幻觉。

## 5. 开发迭代计划（建议 6 个 Sprint）

## Sprint 0（1 周）：需求澄清与技术基线
**目标**：把问题定义清楚，建立可开发骨架。  
**交付**：
- PRD v1（输入字段、输出格式、约束）
- 技术方案（LangGraph + LLM + 检索）
- 项目目录初始化、配置管理、日志体系

## Sprint 1（1~2 周）：信息抽取与追问闭环（MVP 核心）
**目标**：能稳定拿到完整用户画像。  
**功能**：
- 结构化抽取（学校/GPA/预算/国家/专业/入学季）
- 缺失字段识别与追问
- 多轮会话状态维护
**验收标准**：
- 20 组样例对话中，关键字段完整率 ≥ 90%
- 预算标准化正确率 ≥ 95%

## Sprint 2（1~2 周）：院校推荐与概率分层
**目标**：从“问清楚”升级到“给建议”。  
**功能**：
- 引入学校与项目样本库
- 基于规则 + 模型给出冲刺/匹配/保底分层
- 输出推荐理由与费用估算
**验收标准**：
- 每次输出至少 6 所院校（2/2/2）
- 结果包含可解释依据（GPA区间、预算、项目特征）

## Sprint 3（1 周）：时间轴与行动清单
**目标**：方案可执行。  
**功能**：
- 按申请季自动生成月度时间轴
- 生成短期行动清单（未来 2 周）
- 风险节点提醒（语言成绩、DDL）
**验收标准**：
- 时间轴覆盖率 100%（准备→申请→录取→签证）
- 行动项具备“时间+任务+产出物”

## Sprint 4（1~2 周）：RAG 与实时政策增强
**目标**：方案更“新”、更可信。  
**功能**：
- 接入政策检索（Exa + 本地政策知识库）
- 引用来源片段生成事实依据
- 检索失败降级策略（仅本地知识库）
**验收标准**：
- 有检索结果时，报告引用 ≥ 2 条来源
- 检索超时不阻断主流程（可降级）

## Sprint 5（1 周）：评测、稳定性与发布
**目标**：可上线试运行。  
**功能**：
- 建立离线评测集（50~100 样本）
- 质量评测：完整性、可执行性、事实一致性
- API 封装与部署（FastAPI + Docker）
**验收标准**：
- 平均响应时间 < 8s（不含慢检索）
- 严重幻觉率 < 5%
- 核心流程 E2E 测试通过

## 6. 技术架构建议

### 6.1 编排层
- 使用 LangGraph 管理节点：`extract -> clarify -> retrieve -> plan -> evaluate`。
- 每轮执行后写入会话状态，支持连续对话。

### 6.2 模型层
- 抽取任务：低温结构化输出模型。
- 规划任务：中低温长文本模型。
- 评估任务：独立审校模型或同模型二次校验。

### 6.3 数据层
- 学校项目库（结构化）+ 政策库（非结构化）。
- 统一字段字典，避免不同来源口径冲突。

### 6.4 可观测性
- 记录字段缺失率、追问轮次、方案生成耗时、用户采纳率。
- 对失败样本进行归档用于提示词迭代。

---

## 7. 风险与应对
- **风险1：用户输入模糊、口语化严重** → 增强标准化器 + few-shot 抽取样例。
- **风险2：录取概率不可信** → 先规则分层，再让 LLM 解释，不直接“拍脑袋概率”。
- **风险3：政策实时性不足** → 检索 + 本地快照双通道，报告标注“更新时间”。
- **风险4：对话过长成本上升** → 摘要压缩会话，保留关键画像字段。
